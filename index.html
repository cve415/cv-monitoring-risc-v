8 Companies Shaping RISC-V Observability

Many fond memories in performance, monitoring, and observability were during my years at New Relic and AppDynamics watching developers solve issues across front-end and server code. Usually, the culprit was memory bandwidth saturation, cache thrashing, or interconnect contention happening outside the core—things that used to be difficult to see until it was too late.

RISC-V's new Uncore PMU proposal (shoutout to Zhou Song and the team at Alibaba, Hangzhou, China) tackles exactly this blind spot. It standardizes performance monitoring for last-level cache, memory controllers, coherent interconnects, and I/O subsystems—the places where modern AI, HPC, and data center workloads actually live or die.

Why this matters for three different audiences:

🔧 For Developers

You'll get the same uncore visibility Intel and ARM engineers already take for granted. No more assuming 40% CPU utilization means headroom, when your memory bandwidth is actually maxed out and your LLC is thrashing. The fix shifts from "tune the scheduler" to "fix the data layout"—but only if you can see it.

🐧 For Open-Source Advocates

This isn't a vendor-specific hook. It's a shared standard integrated with Linux perf and existing tooling, meaning RISC-V implementations won't fragment into proprietary monitoring silos. Ecosystem cohesion beats ecosystem lock-in.

📈 For Business / GTM

This is infrastructure-level table stakes for cloud and HPC adoption. Customers evaluating RISC-V against x86 or ARM will ask: "Can your platform debug the bottlenecks that actually matter?" This proposal gets us to "yes."

Modern performance problems aren't in the core. RISC-V is finally making the uncore observable—and these are the companies building that future.
